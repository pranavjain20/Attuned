# Session — Mar 18, 2026 (Day 4 Pre-Work: BPM Experiments)

This session was supposed to be the start of Day 4 — LLM song classification for 669 songs. Instead, I spent ~6 hours going deep on BPM accuracy, because the foundation has to be right before building on it.

## The problem

Essentia's classical BPM estimator — the thing I was planning to use as the primary BPM source for song classification — doesn't work on Indian music. Out of 24 songs with confirmed audio files, it got 8 within ±5 BPM. Break it down by category and the picture is worse: 4/5 English songs correct, but only 3/13 Bollywood songs. These aren't octave errors where it detects 160 instead of 80. These are fundamental misdetections — Pal Pal Dil Ke Paas (actual 117 BPM) detected as 153. The autocorrelation-based beat tracker gets confused by tabla patterns, vocal ornaments, and the rhythmic complexity of Indian film music. My library is majority Bollywood. This wasn't going to work.

## Strategy D: solving the audio source problem

Before attacking BPM, I had to make sure I was analyzing the right audio. The earlier experiments (Strategy A) showed that naive YouTube search sometimes downloads covers, remixes, or wrong versions. Strategy D fixes this: search YouTube for the song, get the top 5 results, check each result's duration against Spotify's `duration_ms`, and pick the one within ±15 seconds. If nothing matches, the song fails cleanly instead of analyzing garbage audio.

Result: 24/25 correct audio files. The one miss (Quit Playing Games by Backstreet Boys) had no YouTube result within duration tolerance — probably a live version vs studio version mismatch. The audio source problem is solved.

## TempoCNN: the neural dead end

My first instinct for fixing BPM was to try a neural model. Essentia ships TempoCNN — a convolutional neural network trained on tempo estimation. 115MB model file, requires `essentia-tensorflow`, needs audio at 11025 Hz instead of 44100 Hz. Set it all up, ran it on the same 24 songs.

Result: 8/24 within ±5 BPM. Identical to classical Essentia. Same songs fail, same songs succeed. The neural model was trained on the same Western music corpus, so it has the same blind spots. A fancier algorithm doesn't help when the training distribution doesn't cover your data. Dropped TempoCNN — 115MB of dependency for zero improvement.

## GPT-4o-mini: the surprising contender

On a whim, tried asking GPT-4o-mini to estimate BPM. One API call, all 25 songs, structured JSON output. No audio, no signal processing — just the model's knowledge of these songs from its training data.

Result: 8/25 within ±5 BPM. Same raw accuracy as Essentia, but here's the key — it fails on *different* songs. Essentia nails English pop but misses Bollywood. The LLM misses some English songs but catches Bollywood songs that Essentia can't touch. Combined coverage (best of either): 14/25.

This was the insight that changed the strategy. It's not about finding one perfect BPM method. It's about routing to the method that works best for each song's genre.

## The model shootout

If GPT-4o-mini is good, surely GPT-4o or Claude is better? Tested four more models on the same 25 songs:

- GPT-4o: 6/25 (worse than mini)
- GPT-4.1: 2/25 (dramatically worse)
- Claude Sonnet: 7/25
- Claude Opus: 5/25

Counterintuitive but makes sense once you think about it. BPM isn't a reasoning task — it's a recall task. These songs exist in music databases (Tunebat, Musicstax, SongBPM). The model either memorized the value during training or it didn't. Smaller models that memorize database values outperform larger models that try to "reason" about tempo from genre conventions and mood. GPT-4.1 was the worst because it kept reasoning its way to plausible-but-wrong answers: "this is a romantic Bollywood ballad, so probably 70-80 BPM" when the actual tempo is 117.

## Prompt engineering

Ran three prompt variants against the GPT-4o-mini baseline:

**V1 — Calibrated:** Added duration hints, calibration examples with known BPM, explicit warning about confusing mood with tempo, instruction to consider drum patterns. Result: 7/25. The calibration examples caused overcorrection — the model started anchoring to the example BPMs instead of recalling actual values.

**V2 — Chain of thought:** Forced step-by-step reasoning (identify genre → describe drum pattern → reason about BPM → give estimate). Result: 8/25. No improvement. The reasoning steps added noise without adding information — the model doesn't actually know the drum pattern, it's just generating plausible text.

**V3 — Database recall:** Reframed the task entirely. Instead of "estimate the BPM," asked the model to "recall the precise BPM as documented in music databases like Tunebat." Added half-time/double-time awareness. More calibration anchors but framed as reference data, not correction. Result: 9/25, MAE 13.1 (vs 18.8 baseline).

The framing matters more than the instructions. "Estimate" tells the model to reason. "Recall from databases" tells it to retrieve. For factual recall, retrieval framing wins.

## The external API hunt

Before committing to the LLM approach, tried to find a BPM database API:

**Soundcharts:** Has 1,000 free API requests and comprehensive metadata. Signed up, got API key, started testing. Audio features (including BPM) are locked behind the premium tier — $250+/month. The free tier only gives chart/social data.

**GetSongBPM:** Has an API documented online. Every request returns a Cloudflare challenge page. Tried different user agents, headers, delays. Completely blocked for programmatic access.

**SongBPM.com:** Scraped a few pages. 48% coverage for my test songs. 0% accuracy on the Bollywood songs it did cover — it's clearly using the same Essentia-style analysis under the hood.

**HuggingFace Spotify dataset:** Found a dataset claiming Spotify audio features for millions of tracks. But Spotify's deprecated audio features API was built on... Essentia. The dataset's BPM values have the same failures as running Essentia directly. It's the same data with extra steps.

No free BPM database covers Indian music accurately. This confirmed the LLM hybrid approach as the right call.

## Tolerance analysis

All this work for ±5 BPM accuracy, but does it actually matter? Looked at the state mapper ranges I'll be building in Day 5:

- "Calming/grounding" = 60-80 BPM
- "Gentle recovery" = 70-90 BPM
- "Steady moderate" = 85-110 BPM
- "Energizing" = 100-140 BPM

These ranges are 20-60 BPM wide. A ±10 BPM error on any song would never cause it to land in the wrong state bucket. Even the worst-case scenario — a song at a range boundary with a 10 BPM error — still falls within the overlap zone between adjacent ranges.

This reframed the whole problem. ±5 BPM is nice-to-have. ±10 BPM is all that's required for correct playlist generation. And the hybrid approach (Essentia for English, LLM for Indian) gets us well within that.

## The decision

**Production BPM strategy:** GPT-4o-mini with database-recall prompting + Essentia hybrid.

- For English/Western songs: trust Essentia's BPM (4/5 accuracy at ±5)
- For Indian/Bollywood songs: trust GPT-4o-mini's database-recall BPM
- Genre routing based on artist/language metadata already in the DB
- ±10 BPM tolerance for matching engine
- TempoCNN dropped entirely (115MB dependency, identical accuracy)

Cost: ~$0.02 per 25-song batch for the LLM calls. Negligible.

Removed TempoCNN code and model file (115MB). Cleaned up leftover worktrees from the model shootout experiments. Fixed `.gitignore` to cover all test audio directories.

## Reflection

The interesting lesson from today isn't technical — it's about assumptions. I started this session assuming that audio analysis is the "right" way to determine BPM and that LLMs would be a fallback. Six hours later, the LLM is the primary strategy for most of my library, and audio analysis is the fallback for the subset of music where signal processing happens to work.

The assumption was that BPM estimation is a signal processing problem. It's not — at least not for a personal music library that's 60% Bollywood. It's a data retrieval problem. These songs have known BPMs. They're documented in databases. The question isn't "how do I compute the BPM from audio?" It's "how do I look up the BPM that someone already computed?" And it turns out a small language model that memorized music database entries during training is a better lookup table than a sophisticated signal processing pipeline.

There's a broader pattern here about non-Western content and ML tooling. Essentia is an excellent library, well-maintained, widely used. TempoCNN is published research. But both were trained on Western music corpora. The autocorrelation assumptions that work for kick-snare patterns fall apart on tabla. The neural network that learned tempo from pop/rock/electronic can't transfer to Bollywood. And the "database" solutions (Spotify features, SongBPM, HuggingFace datasets) all trace back to the same Western-biased analysis. If your content isn't in the training distribution, more sophisticated processing of the same biased data doesn't help. You need a fundamentally different data source.

GPT-4o-mini works because it saw Tunebat and Musicstax pages during training. Those databases were built by humans who actually listened to the songs. Human annotation, at scale, captured through web scraping and compressed into model weights. Sometimes the least technical solution is the most robust one.

# Session Part 2 — Property-by-Property Evaluation (remaining 5 properties)

After BPM, I went through each remaining Essentia property using the same process: understand the algorithm, run on the 24 test files, compare against ground truth, diagnose failures, try alternatives, decide. Here's what happened.

## Key/Mode — Essentia is good enough

**Algorithm:** `es.KeyExtractor()` computes an HPCP (Harmonic Pitch Class Profile) — a 12-dimensional vector of how much energy exists at each pitch class. Then it matches against templates for all 24 key/mode combinations and picks the best fit.

**Ground truth:** Looked up Tunebat/SongBPM data for all 25 test songs. Critical caveat: these databases source their values from Spotify's deprecated audio features API, which itself runs Essentia on official audio masters. So "ground truth" is actually Essentia-on-Spotify vs our Essentia-on-YouTube.

**Results:**
- 12/24 exact match (50%)
- 2/24 enharmonic match (C# = Db) = effectively exact
- 2/24 parallel key (same root, wrong major/minor)
- 2/24 relative key (musically compatible — different root, same key signature)
- 6/24 completely wrong (25%)

**What the "wrong" results actually are:** Analyzed the 6 failures musically. 3 were dominant confusion (Essentia detected the 5th instead of the root — e.g., detected E major when the song is in A major). This is a known HPCP limitation: the V chord is the most common chord after I, so their HPCP profiles are similar. 1 was subdominant confusion (4th instead of root). Only 2 were truly unrelated keys.

**Why this matters for sequencing:** The matching engine uses key for song ordering within playlists (smooth transitions via Camelot wheel). Dominant and subdominant keys ARE compatible on the Camelot wheel — they're 1 step apart. So the effective accuracy for our use case is 22/24 (92%).

**What I tried beyond default:**
- EDMA profile (tuned for electronic dance music analysis)
- Bgate profile (different weighting)
- All 3 profiles agreed 21/24 times — the algorithm is internally consistent, just sometimes wrong

**Decision:** Essentia KeyExtractor is sufficient. 92% musically usable for sequencing. The 8% failures are concentrated in old Bollywood (raga-based music that doesn't map cleanly to Western 12-key tonality). No code changes needed.

**Accuracy:** 58% exact, 67% exact+enharmonic, 75% musically close, 92% usable for sequencing.

## Energy — Fix the normalization, don't overthink it

**Algorithm:** `RMS(audio) / 0.25`, clamped to [0,1]. RMS = root mean square amplitude = average loudness.

**The problem:** 11/24 songs (46%) clipped at 1.0. The 0.25 divisor was too low. Everything from "Levitating" to "Apna Bana Le" (a romantic ballad) scored identically at 1.0. Zero discrimination in the top half of the library.

**What I tried:**
1. **RMS/0.35** — Fixes the ceiling: 0 songs at 1.0. 71% bucket accuracy (low/medium/high). This was the best.
2. **Log-normalized Loudness (EBU R128)** — Better discrimination (range: 1,873 to 12,646) but only 1 song at ceiling. Rankings mostly reasonable. Correlation with RMS: 0.754.
3. **Percentile-based normalization** — Rank each song's RMS within the test set. 62% accuracy. Worse because it spreads the distribution too evenly.
4. **Sigmoid normalization** — S-curve centered on median RMS. 67% accuracy. Same spreading problem.
5. **90th-percentile frame energy** — Instead of average RMS, use the loudest 10% of frames. 67% accuracy. Mastering compression reduces frame variation in modern music.
6. **P90 * (1 + coefficient of variation)** — Peak energy weighted by rhythmic variation. 58%. Worse.
7. **Composite: 35% loudness + 25% spectral brightness + 20% onset rate + 20% BPM** — Tried combining four features that should theoretically predict energy. Rankings were terrible: Tum Se Hi (slow ballad) at #3, Levitating at #16. Each component has confounds (brightness ≠ energy, onset rate ≠ energy, wrong BPMs corrupt the signal). The composite amplified noise instead of canceling it.

**Key insight:** Energy is a perceptual property — how intense a song *feels*. It combines loudness, dynamics, timbre, rhythmic intensity, and genre context. No single acoustic measurement (or simple combination) captures this. RMS measures volume, which is ~70% correlated with perceived energy. That's the best we can get from signal processing without trained ML models.

**The founder mindset realization:** Every complex alternative scored worse than the one-line fix (changing 0.25 to 0.35). Simple approaches beat complex ones when the underlying signal is weak. Save composites for when individual components are >85% accurate.

**Decision:** RMS/0.35. One-line change, 71% bucket accuracy, no API cost. The matching engine's other signals (BPM, danceability, acousticness) compensate for the 29% noise.

**Accuracy:** 71% three-bucket classification (low/medium/high).

## Danceability — Essentia is broken, use LLM

**Algorithm:** `es.Danceability()` uses DFA (Detrended Fluctuation Analysis) — a fractal analysis technique that measures rhythmic self-similarity. Higher DFA = more predictable/regular rhythm = theoretically more danceable.

**The problem:** 23/24 songs scored above 1.0 and were clamped to 1.0. ONE song (Pal Pal Dil Ke Paas) scored 0.995. Every other song — from a Sufi prayer to an EDM-influenced Punjabi banger — got exactly 1.0. Zero discrimination.

**What I tried:**
- Rescaling the raw range [0.995, 1.486] to [0,1]: gave 42% bucket accuracy (worse than random guessing at 33% for 3 buckets)
- The rescaled ranking was clearly wrong: BSB "Quit Playing Games" (slow 90s ballad) ranked #4 most danceable, Levitating ranked #17

**Why it can't be fixed:** DFA measures rhythmic REGULARITY, not danceability. A metronome is perfectly regular but not danceable. A slow ballad with steady 4/4 time (like BSB) scores high regularity. The algorithm measures the wrong thing entirely. No normalization, rescaling, or threshold tuning changes what the measurement captures.

**Decision:** Drop Essentia for danceability. LLM only. No extra API cost since we're already calling GPT-4o-mini for BPM.

**Accuracy:** 42% (worse than random). Unusable.

## Acousticness — Switch from spectral centroid to spectral flatness

**Algorithm (before):** Inverted SpectralCentroidTime. `1.0 - (sct - 500) / 1500`. Higher centroid = brighter sound = less acoustic.

**The problem:** 33% bucket accuracy — random guessing. Spectral centroid measures BRIGHTNESS, not acousticness. A bright acoustic guitar (high centroid) scores "electronic." A bass-heavy EDM track (low centroid) scores "acoustic." Chaand Taare (old Bollywood classic with tabla and strings) scored 0.0 acousticness. Cheques (modern electronic Punjabi) scored 0.72 acousticness. Exactly backwards.

**What I tried:**
1. **Spectral Centroid (current):** 33% — random
2. **Spectral Flatness:** 67% — significantly better. Flatness measures how "tonal" vs "noise-like" the spectrum is. Acoustic instruments produce clean harmonic overtones (low flatness). Electronic production adds compression, distortion, and synthetic timbres that make the spectrum noisier (high flatness). The inversion maps correctly: low flatness → high acousticness.
3. **Spectral Rolloff:** 33% — same as centroid, no better

**Why flatness works better:** Tonal purity is a better indicator of organic instruments than brightness. A piano, guitar, violin — these produce clear harmonic series. A synthesizer, drum machine, or heavily compressed mix produces a flatter, noisier spectrum. The correlation between flatness and "organic vs synthetic" isn't perfect (bass-heavy clean synths fool it), but it's directionally correct.

**Code change:** Replaced the single-line spectral centroid inversion with frame-level spectral flatness computation. Iterates over audio frames, computes flatness per frame, averages. Maps with `1.0 - avg_flatness * 20`.

**Decision:** Spectral flatness replaces spectral centroid. 67% bucket accuracy — same tier as energy (71%).

**Accuracy:** 67% three-bucket classification (high/medium/low acoustic).

## Instrumentalness — ZCR is broken, use LLM

**Algorithm:** Zero Crossing Rate scaled to [0,1]. `(zcr - 0.02) / 0.08`. Theory: vocals are mid-frequency, so vocal tracks have lower ZCR than instrumental tracks with more high-frequency content (cymbals, distorted guitars).

**The problem:** 11/24 vocal songs scored above 0.5 instrumentalness. Abhijeet's "Chaand Taare" scored 0.88 (pure vocal track). BSB's "Quit Playing Games" scored 0.75. The entire test set is vocal tracks — instrumentalness should be near 0 for all of them.

**Why it can't be fixed:** ZCR measures high-frequency spectral content, which correlates with production style (bright mastering, cymbal-heavy mixes) not with vocal presence. Detecting whether a song has vocals requires understanding what "voice" sounds like — a pattern recognition problem that needs ML models (Essentia's TensorFlow-based voice detection) or semantic knowledge (LLM knows whether "Chaand Taare" has vocals).

**What I considered:** Essentia has TF-based voice detection models, but they require `essentia-tensorflow` (heavy dependency, same training bias we rejected with TempoCNN). Also, all 669 songs in the library have vocals — instrumentalness has limited matching value for this specific use case.

**Decision:** Drop Essentia for instrumentalness. LLM only.

**Accuracy:** Not meaningful — the proxy measures the wrong thing.

## Valence, Mood Tags, Genre Tags — LLM only

These three properties were never expected to come from Essentia — they're semantic/perceptual by nature. Valence (positive vs negative emotional feel), mood (chill, uplifting, melancholic), and genre (Bollywood, pop, Punjabi) require understanding musical context, not measuring acoustic properties.

**Decision:** LLM only (by design).

## The meta-lesson

The biggest takeaway isn't about any specific property — it's about the difference between acoustic measurement and perceptual judgment.

Essentia is excellent at what it does: measuring acoustic properties of audio. RMS is the correct root mean square. Flatness is the correct spectral flatness. DFA is the correct detrended fluctuation analysis. The algorithms work.

The problem is that most of the properties we need for playlist matching aren't acoustic properties. "Energy" isn't loudness — it's how intense a song *feels*. "Danceability" isn't rhythmic regularity — it's whether people would dance to it. "Acousticness" isn't spectral purity — it's whether the instruments are organic. These are human judgments that require context: genre conventions, cultural expectations, production norms.

Where acoustic measurements do work (BPM for Western music, key detection, relative loudness, spectral purity), Essentia is solid and free. Where they don't, the LLM's training-data-encoded knowledge of how music is perceived is fundamentally more accurate than signal processing — because the LLM learned from human descriptions of music, not from waveform analysis.

The practical answer isn't "Essentia vs LLM." It's "use each for what it's actually good at." Essentia for objective acoustic properties. LLM for subjective perceptual properties. This isn't a compromise — it's the correct architecture.

## The LLM energy/acousticness test — assumptions overturned

After completing the property evaluation, I was confident that LLM would beat Essentia for energy and acousticness. My reasoning: "the LLM knows what 'energetic' means semantically, so it should classify energy better than a crude RMS measurement." I estimated 85-90% accuracy for the LLM.

Pranav pushed back: "Test it first." So I ran GPT-4o-mini on the same 25 test songs for energy and acousticness, using the same expected labels.

Results:
- **Energy:** LLM 42% vs Essentia 71%. Essentia wins by 29 points.
- **Acousticness:** LLM 50% vs Essentia 62%. Essentia wins by 12 points.

The LLM was dramatically worse. Why?

**Middle-value bias.** The LLM scored almost every song between 0.4 and 0.8. It rarely gave values below 0.3 or above 0.9. This is a known tendency — when asked to rate something on a continuous scale, LLMs hedge toward the middle. Kun Faya Kun (a quiet Sufi prayer) got energy 0.80. Namo Namo (an acoustic devotional song) got acousticness 0.40 (low acoustic). These aren't close calls — they're confidently wrong.

**Weak knowledge of Indian music.** The LLM correctly identified Levitating as high energy and low acoustic. It got the Western songs mostly right. But for Bollywood, it guessed poorly — likely because its training data has fewer detailed descriptions of Indian film songs than Western pop.

**Audio analysis beats metadata guessing.** Essentia's RMS measurement is crude, but it's measuring something real — the actual loudness of the audio signal. The LLM is guessing from the song title and artist name. When the song is well-known in Western music databases (Levitating, Watermelon Sugar), the LLM does fine. When it's not (Namo Namo, Kun Faya Kun), it falls back to genre-based assumptions that are often wrong.

**The lesson:** Don't assume LLMs beat specialized tools for continuous-value predictions. LLMs excel at categorical judgments (is this song danceable? yes/no), mood tagging (what mood is this?), and genre classification (what genre?) — tasks where they can recall training data directly. But for continuous 0-1 scales where precision matters, they compress toward the middle and lose to even crude measurements that engage with the actual signal.

This changed the final architecture. Energy and acousticness stay with Essentia.
